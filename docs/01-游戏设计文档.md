# Kpop嫂子模拟器 — 游戏设计文档

> 版本：v1.0 | 最后更新：2026-06-06

---

## 一、游戏概述

**游戏名称**：Kpop嫂子模拟器

**游戏简介**：一款以Kpop娱乐圈为背景的恋爱模拟网页游戏。玩家扮演自定义女主角，在回合制行程安排中与爱豆发展关系，经历攻略→恋爱→结婚三个阶段，面对现实压力和道德选择，最终走向不同的结局。

**核心体验**：沙盒式自由体验，无"正确"玩法——每个玩家的故事都由自己的选择塑造。

---

## 二、游戏类型与表现形式

| 维度 | 决定 |
|------|------|
| **游戏类型** | 回合制行程安排 + 选择驱动 + 多结局 |
| **平台** | 移动端网页（响应式，主适配320px-430px） |
| **UI框架** | 底部五卡片导航（日程/手机/我的/关系/事件） |
| **回合单位** | 半个月 |
| **总时长** | 2-3年（48-72回合），事件驱动结束 |
| **技术栈** | 纯 HTML/CSS/JS + DeepSeek API |

## 三、核心基调与目标

| 维度 | 决定 |
|------|------|
| **核心哲学** | 提供有意义的选择，不引导"完美路线" |
| **胜负标准** | ❌ 无固定标准，所有结局都是"你的故事" |

## 四、女主人设

| 维度 | 决定 |
|------|------|
| **姓名** | 玩家自由输入 |
| **身份** | 预设标签可选 + 自定义输入 |
| **性格/人设** | 预设标签可选 + 自定义输入 |

### 身份标签预设
素人粉丝 / 练习生 / 造型师 / 记者 / 翻译 / 站姐 / 综艺PD / 青梅竹马 / 学生 / 化妆师 / 运动教练 / 名门社交好友 / 其他（自定义）

### 性格标签预设
直球勇敢 / 温柔体贴 / 傲娇高冷 / 腹黑心机 / 社恐慢热 / 其他（自定义）

### 身份对数值的影响
不同身份影响好感度加成和嫌疑度判定：
- 粉丝/站姐：追星场景好感+，但被拍到→嫌疑度暴涨
- 练习生/同行：工作场景好感+，同框合理→嫌疑度低
- 青梅竹马/同学：天然信任基础→初期嫌疑度极低
- 工作人员：专业场景好感+，合理接触多→嫌疑度低
- 名门/社交圈：社交场合合理→嫌疑度看场合

## 五、爱豆人设

| 维度 | 决定 |
|------|------|
| **设计方式** | 原创Kpop风爱豆 + 玩家完全自定义 |
| **姓名** | 自由输入 |
| **性别** | 自由选择（男/女） |
| **性格/人设** | 预设标签可选 + 自行输入描述 |
| **所属团体** | 自由输入团名 |
| **人数** | 玩家自行决定攻略几个爱豆 |
| **备注/爱称** | 可设置，系统使用原名 |

### ⚠️ 性别约束
所有后续剧情/对话/事件/AI生成内容必须严格匹配玩家设定的性别：
- 数据模型：每个爱豆存储 `gender` 字段
- AI对话：系统提示词明确传入性别，使用正确代词
- 事件文本：使用变量（如`{他/她}`、`{哥哥/姐姐}`）而非写死
- 不得出现"设定女爱豆但剧情套用男爱豆模板"的情况

## 六、核心数值系统

| 数值 | 范围 | 说明 |
|------|------|------|
| ❤️ 好感度 | 0-100/每人 | 分爱豆独立计算，受身份标签加成/减成 |
| 😰 压力值 | 0-100 | 过高触发负面事件（崩溃、失误） |
| 🕵️ 嫌疑度 | 0-100 | 恋情暴露风险，不同身份下判定不同 |
| ⚡ 体力 | 0-100 | 每回合行动消耗，休息恢复 |
| 💎 魅力 | 0-100 | 影响初始好感、事件触发门槛、对话成功率 |

### 显示方式
所有数值**全可见**，以进度条/数字形式展示。

### 关系阶段门槛
| 阶段 | 好感度要求 | 说明 |
|------|-----------|------|
| 💫 攻略期 | 0-32 | 默认阶段，可多人并行攻略 |
| 💕 恋爱期 | 33+ | 达到33%触发确认关系事件 |
| 💍 结婚 | 90+ | 达到90%触发求婚事件 |

> **设计说明**：结婚门槛设为90%而非66%，体现"结婚是重大决定"的真实感。恋爱门槛33%保持较低，让玩家早期就能体验关系推进。

## 七、主要玩法模块

### 7.1 回合制核心循环

```
每回合开始
  ├── 📋 查看当前数值面板
  ├── 🎯 选择行动（消耗体力）
  │     ├── 💬 和爱豆互动（发消息/约会/探班/…）
  │     ├── 🏋️ 提升自己（健身/学才艺/护肤/…）
  │     ├── 📱 经营社交（发SNS/和粉丝互动/…）
  │     ├── 💼 工作/学业
  │     ├── 🎁 送礼/应援
  │     ├── 🔍 收集情报
  │     └── 🏥 休息（恢复体力、降压）
  ├── 🎲 触发事件（如有）
  ├── 📊 数值变动结算
  └── 📖 推进剧情（如达到条件）
```

### 7.2 行动系统
- 不同身份/人设解锁不同行动子选项
- 每个行动消耗体力，不同行动消耗量不同
- 不同选项影响不同数值

### 7.3 底部五卡片导航

| 选项卡 | 功能 |
|--------|------|
| 📅 日程 | 每回合的行动安排 |
| 📱 手机 | 核心社交工具（聊天/SNS/新闻/通话/购物） |
| 👤 我的 | 查看女主数值、身份、属性面板 |
| 💜 关系 | 查看各爱豆好感度、关系状态、嫌疑度 |
| 🗂️ 事件 | 已触发事件回顾、剧情日志、结局回顾 |

## 八、手机系统

### 8.1 手机内软件

| 软件 | 功能 | 互动对象 |
|------|------|---------|
| 💬 聊天APP | 私聊 | 一对一（和爱豆） |
| 📞 通话 | 语音通话（重要/紧急） | 一对一 |
| 📸 SNS/社交媒体 | 发动态、看动态、互动 | 公开/半公开 |
| 📰 新闻/热搜 | 娱乐新闻、绯闻、回归消息 | 被动接收 |
| 🛒 购物/应援 | 买礼物、应援物、服饰 | 系统交互 |
| 📧 邮件 | 工作/学校通知 | 系统/工作 |

### 8.2 多手机机制
- **主手机**：日常使用，所有联系人可见
- **秘密手机**：剧情触发后获取（经纪人给/自己买）
- 切换方式：手机顶部标签或下拉菜单
- 限制：最多2-3台手机
- 风险：秘密手机被发现→嫌疑度暴涨
- 视觉区分：主手机亮色UI / 秘密手机暗色UI

### 8.3 对话系统
- 预设选项（2-4个）+ 自定义输入框
- 爱豆聊天头像可上传，未上传则不显示（仅气泡）
- 玩家手机壁纸可自定义
- 对话历史本地存储

## 十、AI接入（DeepSeek API）

| 维度 | 决定 |
|------|------|
| **API提供商** | DeepSeek |
| **Key管理** | 玩家自备API Key |
| **应用场景** | 对话、SNS动态、新闻、随机事件、关键剧情、结局文本、身份buff/debuff分析 |
| **调用策略** | 每次"对话回合"调一次；预设选项可预写，自定义输入走API |
| **上下文** | 每次调用传入当前数值/身份/关系阶段（结构化状态注入，见下文） |

### 10.1 身份Buff/Debuff分析

**问题**：玩家可自定义职业/身份（如"大学生"、"护士"、"自由职业者"），系统无法预设所有职业的数值修正。

**方案**：由DeepSeek API根据职业描述，自动生成合理的buff/debuff数值。

**流程**：
1. 玩家在设定阶段输入自定义身份
2. 游戏开局后，将该身份文本发送给DeepSeek API
3. API返回结构化JSON，包含各项数值修正（范围-20 ~ +20）
4. 存入 `Game.state.player.identityModifiers`，后续所有数值变化自动应用

**Prompt示例**：
```
你是一个游戏数值设计师。根据以下职业/身份，为5个维度生成数值修正（范围-20到+20）：
- 好感度加成（affectionBonus）：该身份在恋爱互动中的优势
- 嫌疑度修正（suspicionMod）：该身份被怀疑的风险（正数=更易被怀疑）
- 体力消耗修正（staminaMod）：该身份日常体力消耗（正数=消耗更多）
- 魅力加成（charmBonus）：该身份对异性的吸引力
- 压力抗性（stressResist）：该身份应对压力的能力

职业：大学生
返回纯JSON（不要markdown包裹）：
{"affectionBonus": ..., "suspicionMod": ..., "staminaMod": ..., "charmBonus": ..., "stressResist": ..., "reasoning": "简短分析"}
```

**降级方案**（API不可用时）：使用预设默认值表，覆盖常见职业。

### 10.2 上下文窗口管理策略

**问题**：游戏可能持续48-72回合，每回合多次对话。如果将所有历史对话都发给DeepSeek，很快就会超出128K上下文窗口，导致API"遗忘"最初的角色设定和关键选择。

**方案**：结构化状态注入 + 滑动对话窗口 + 长短期记忆

#### 10.2.1 三层上下文架构

```
┌─────────────────────────────────────────────┐
│ Layer 1: 固化状态（始终注入，~5K tokens）      │
│ 玩家身份+性格 / 爱豆全量数据 / 五大数值 /      │
│ 关系阶段+好感度 / 身份修正值 / 关键里程碑       │
├─────────────────────────────────────────────┤
│ Layer 2: 长期记忆（始终注入，~2K tokens）      │
│ 重要选择记录 / 告白/分手等关键事件 / 纪念日     │
│ 格式：一句话摘要 + 回合数                      │
├─────────────────────────────────────────────┤
│ Layer 3: 近期对话（滑动窗口，~30K tokens）     │
│ 最近10-15条对话记录（含AI回复）                │
│ 超出窗口的对话 → 摘要后移入Layer 2             │
└─────────────────────────────────────────────┘
```

#### 10.2.2 具体机制

| 机制 | 说明 |
|------|------|
| **状态摘要注入** | 每次API调用前，自动拼装当前游戏状态的JSON摘要作为system prompt前缀 |
| **对话滑动窗口** | 每个爱豆保留最近N条完整对话（建议10-15轮），旧对话压缩为1-2句摘要 |
| **关键事件标记** | 告白、确定关系、求婚、分手等重要事件永久保留在Layer 2 |
| **数值变化日志** | 不在API上下文中发送完整变化历史，仅在状态摘要中体现当前值 |
| **Token预算预估** | 每次调用前检查上下文长度，超出预算则自动裁剪Layer 3 |

#### 10.2.3 上下文预算分配

| 层 | 预算 | 内容 |
|----|------|------|
| System Prompt | ~3K | 游戏规则 + 输出格式要求 |
| Layer 1 固化状态 | ~5K | 当前游戏状态JSON |
| Layer 2 长期记忆 | ~2K | 关键事件列表 |
| Layer 3 近期对话 | ~30K | 最近N轮对话 |
| AI回复预留 | ~5K | 输出token预算 |
| **总计（安全线）** | **~45K** | 远低于128K上限 |

#### 10.2.4 对话摘要触发条件

当某爱豆的对话记录超过滑动窗口上限时：
1. 取最旧的一半对话 → 调用DeepSeek生成1-2句摘要
2. 摘要存入Layer 2："[回合3-8] 和智秀的早期对话：讨论回归准备，玩家表示支持，氛围轻松友好"
3. 完整对话从Layer 3移除
4. 后续API调用中，Layer 2摘要提供"早期记忆"，Layer 3提供"近期细节"

> **设计保证**：无论玩家玩多少回合，DeepSeek永远能"记住"角色的核心设定和当前状态。不会出现"玩到后期，AI忘了玩家是谁"的情况。

| 维度 | 决定 |

| **备选方案** | A. 韩系极简风 |
